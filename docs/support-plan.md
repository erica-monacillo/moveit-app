# Support Plan

## Project

MoveIt App

## Support Process

Users can report issues by creating a GitHub Issue in the project repository.

Each issue should include:

- A clear title
- Description of the problem
- Steps to reproduce
- Expected result
- Actual result
- Screenshot, if available
- Browser or device information

## Issue Reporting Process

1. User finds a problem.
2. User creates a GitHub Issue.
3. The issue is reviewed and labeled.
4. The issue is assigned a priority.
5. A developer investigates the issue.
6. A fix is created on a bugfix branch.
7. A Pull Request is opened.
8. The fix is reviewed and merged.
9. The system is redeployed.
10. The issue is closed after verification.

## Common Issues

| Issue | Possible Cause | First Response |
|---|---|---|
| Page does not load | Deployment failed or service outage | Check deployment status and logs |
| Feature not working | Code bug or missing data | Reproduce the issue and check console |
| Styling looks broken | CSS or build issue | Check build output and browser compatibility |
| Link is broken | Incorrect route or URL | Verify routing and update the link |
| Slow loading | Large assets or network issue | Check performance and optimize assets |

## Escalation

Issues should be escalated when:

- The application is completely unavailable
- A major feature is broken
- Data loss is possible
- A deployment fails repeatedly
- A security concern is reported

## Response Times

| Priority | Example | Response Time |
|---|---|---|
| High | App is unavailable | Within 24 hours |
| Medium | Main feature is broken | Within 2 business days |
| Low | Small visual issue or typo | Within 5 business days |

## Support Roles

| Role | Responsibility |
|---|---|
| Developer | Investigates and fixes issues |
| Reviewer | Reviews Pull Requests |
| Tester | Verifies fixes after deployment |
| Instructor | Provides guidance when escalation is needed |
